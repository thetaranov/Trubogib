2.3. Тестовая программа (Smoke Test)

D = 38 мм, R = 76 мм, ClampLength = 60 мм

Шаг 1: L_feed=100, Rotation=0°,  Angle=90°
Шаг 2: L_feed=200, Rotation=0°,  Angle=0°

Ожидаемый результат:
  Шаг 1: полный 7-фазный цикл гибки (анимация 90°)
  Шаг 2: пропуск гибочных фаз (Angle=0 → только подача)
  L_total ≈ 100 + π×76×90/180 + 200 = 419.38 мм

5. Протокол синхронизации EXE ↔ Teensy ↔ ПЛК

EXE общается с Teensy через USB-Serial (115200, ASCII-команды + JSON-ответы).

Команды EXE → Teensy (текст, завершаются \n):
  ?             — запросить JSON-статус
  SON 1 / SON 0 — включить/выключить серво (CMD 6/7 → Modbus → ПЛК DO7/DO8)
  HZ / HC       — Home ось Z / C
  MZ mm,spd     — движение оси Z (мм, мм/с)
  MC deg,spd    — движение оси C (°, °/с)
  TGT angle     — задать целевой угол гибки
  CMD 4         — запустить цикл гибки (авто)
  CMD 2         — стоп цикла
  CMD 3         — обнулить угол энкодера
  STOP          — остановить программу
  ESTOP         — аварийная остановка

Ответ Teensy → EXE (JSON, каждые 50 мс по команде '?'):
  {"v":"2.1","mb":{"rx":N,"tx":N,"err":N,"link":N},
   "z":{"mm":0.00,"st":0,"son":1},
   "c":{"deg":0.00,"st":0,"son":1},
   "bend":{"cur":0,"tgt":90,"cyc":1,"mode":1,"done":0},
   "ext":{"do7":1,"do8":1},
   "alm":{"z":0,"c":0},
   "wd":0}

Цикл выполнения шага:
  1. EXE: SON 1 → ждёт фактические PLC do7=1 и do8=1
  2. EXE: MZ feed_mm,100 → Teensy генерирует Step/Dir → серво Z
  3. EXE: MC rotation_deg,30 → Teensy генерирует Step/Dir → серво C
  4. EXE: TGT angle + CMD 4 → Teensy пишет Modbus рег.9-10 → ПЛК → VFD
  5. EXE опрашивает статус (?) каждые 50 мс, ждёт bendDone=1
  6. EXE: MZ +clampLength → отвод каретки
  7. EXE: CMD 4 → возврат ролика (цикл 2), ждёт bendDone=1, cyc=1
  8. EXE: MZ -clampLength → возврат каретки
  9. Переход к следующему шагу

6. Режим Dry Run — тестирование без Teensy

Dry Run позволяет проверить всю анимацию и логику EXE без подключённого оборудования. Оси Z и C выполняются виртуально. Ось гибки (ПЛК) — также симулируется.

Как запустить Dry Run:

  1. Запустить TubeBender.exe. В редакторе загрузить или создать программу гибки.
  2. Панель «Станок» → кнопка Dry Run (не требует подключения к оборудованию).
  3. Наблюдать: 3D-анимация показывает полный цикл гибки со всеми 7 фазами.
  4. Проверить: нет коллизий (труба не пересекает станину/оснастку), углы корректны, L_total верная.
  5. Если всё нормально — можно переходить к реальному запуску (кнопка Start).

Dry Run vs реальный запуск:

ПараметрDry RunРеальный запуск (Start)
Ось Z (подача)Виртуально (анимация EXE)Teensy → Step/Dir → серво Z (реальное движение)
Ось C (вращение)Виртуально (анимация EXE)Teensy → Step/Dir → серво C (реальное движение)
Ось гибкиВиртуальноTeensy → Modbus → ПЛК → VFD → мотор
Нужен Teensy?НетДа (USB-Serial)
Нужен ПЛК?НетДа (Modbus RS-485)

7. Полная последовательность запуска комплекса

Шаг 1 — Подготовка железа

  1. Убедиться, что E-Stop нажата (разомкнута). Осмотреть станок.
  2. Включить автомат 380В (питание VFD и сервоусилителей). Подождать 10 сек (заряд конденсаторов).
  3. Включить автомат 24В (питание ПЛК и дискретных цепей).
  4. Разблокировать E-Stop. Убедиться: дисплей серводрайверов показывает d09.
  5. Убедиться: ПЛК работает (индикатор RUN горит зелёным).

Шаг 2 — Подключение Teensy 4.1

WiFi (ESP8266) больше не используется. ESP8266 применялся только для отладки по HTTP. Вся связь теперь через USB-Serial + Modbus RTU.
  1. Подключить Teensy 4.1 к ПК по USB-кабелю.
  2. Убедиться: связь по Modbus RTU с ПЛК установлена (Serial1, 115200 8N1).
  3. Убедиться: EXE видит Teensy через USB-Serial (COM-порт определяется автоматически).

Шаг 3 — Запуск EXE

  1. Запустить v2.1 (C++ OCCT)/build/Release/TubeBender.exe.
  2. Меню «Окна» → «Станок». Выбрать COM-порт Teensy, нажать «Подключить».
  3. Меню «Окна» → «Подключение». Убедиться: все 4 узла зелёные (PC↔Teensy, Teensy↔ПЛК, Серво Z, Серво C).
  4. Нажать «Servo ON». Убедиться: SON Z и SON C = ON (ПЛК DO7/DO8 замкнуты, +24В на CN1 pin 15).

Шаг 3.1 — Если Teensy постоянно отваливается по USB

Быстрый чек-лист:
  • Общий 0В: Teensy GND, MAX485/MAX490 GND и 0В БП управления должны иметь общий reference.
  • PE не использовать как сигнальный «минус».
  • USB-кабель короткий, экранированный, желательно с ферритом.
  • Разнести силовые кабели VFD/серво и USB/RS-485 по трассам.
  • На входы Teensy нельзя подавать >3.3В (по спецификации PJRC для Teensy 4.1 — 3.3V only).

Шаг 4 — Dry Run (обязательная проверка перед реальным запуском)

  1. Загрузить тестовую программу: Шаг1 (L=100, A=90°), Шаг2 (L=200, A=0°).
  2. Кнопка Проверка коллизий. Убедиться в отсутствии пересечений.
  3. Кнопка Dry Run. Наблюдать полную анимацию. Убедиться в корректности.

Шаг 5 — Реальный запуск

  1. Убедиться: ПЛК в состоянии READY (HR[2]=0 в Modbus).
  2. Труба зажата в патроне. Гибочная оснастка установлена.
  3. Кнопка Start в панели «Станок» EXE.
  4. Наблюдать: 3D-анимация EXE синхронна с реальным движением оси гибки.
  5. Проверить лог: logs/session_YYYYMMDD_HHMMSS.txt — ошибок нет.

Шаг 6 — Проверка аварийных сценариев (обязательно при первом запуске)

  1. Нажать E-STOP в EXE во время цикла → система должна остановиться немедленно (ESTOP → Teensy → Modbus CMD 10 → ПЛК).
  2. Отключить USB-кабель → Modbus watchdog Teensy (2 сек) должен снять Servo-ON и прекратить Step/Dir.
  3. Проверить окно «Подключение»: при потере связи соответствующие узлы становятся красными.
  4. Имитировать аларм серво (например, снять питание 380В при включённом SON) → EXE должен показать «АЛАРМ серво Z/C! (D17/D18)» красным, программа должна перейти в Error.

9. Обработка алармов серво (ALM D17/D18)

Серводрайверы MR-JE-200A сообщают о неисправности через выход ALM (CN1 pin 37, бело-голубой провод). Выход настроен как b-contact (NC): в норме транзистор включён (DI = TRUE), при аварии — выключен (DI = FALSE).

9.1. Цепочка прохождения аларма

MR-JE-200A ALM (CN1 pin 37, b-contact NC, бело-голубой провод)
  → ПЛК D17 (%IX1.1.6) = Серво Z ALM / D18 (%IX1.1.7) = Серво C ALM
  → ПЛК: almZ := NOT gDI_17; almC := NOT gDI_18;
  → Modbus регистр 7 (gAlarmFlags): bit0=almZ, bit1=almC
  → Teensy → JSON: "alm":{"z":0,"c":0}
  → EXE: парсинг alm.z / alm.c

9.2. Реакция EXE на аларм

УсловиеДействие EXE
alm.z = 1 или alm.c = 1 (вне программы)Красная надпись «АЛАРМ серво Z/C! (D17/D18)» в окнах «Подключение» и «Станок»
alm.z = 1 или alm.c = 1 (во время программы)MachPhase → Error. Программа останавливается.

9.3. Действия оператора при аларме

  1. Определить код ошибки на дисплее серводрайвера (AL.xx — см. раздел 03).
  2. Устранить причину аварии (перегрузка, перегрев, обрыв энкодера и т.д.).
  3. Сбросить аварию на серводрайвере (кнопка RES или цикл питания).
  4. Убедиться: DI на ПЛК вернулся в TRUE (D17/D18 = 1), аларм снят в EXE.
  5. Повторить Servo ON и продолжить работу.
