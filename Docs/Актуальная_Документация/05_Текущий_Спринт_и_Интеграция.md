# 05 — Текущий спринт: интеграция EXE + PLC + ESP + серво (без Teensy)

> Консолидированный документ. Источник: `Docs/10_ТЗ_OPUS46_ПОЛНАЯ_ИНТЕГРАЦИЯ.md`.

**Цель этапа:** объединить существующие компоненты в рабочий контур `EXE + PLC + ESP + сервокомплекты`, чтобы запуск гибки из EXE приводил к синхронному реальному движению осей гибки и совпадал с визуализацией. Оси Z/C на этом этапе — Dry Run (виртуально).

## 2. Текущая архитектура (без Teensy)

```
EXE v2.1 (C++ OCCT)
  │ HTTP/JSON (WiFi)
  ▼
ESP8266 (шлюз/промежуточный контроллер)
  │ Modbus RTU RS-485
  ▼

  │
```

**Оси Z и C (текущий этап):** Dry Run. ESP8266 не способен генерировать надёжный Step/Dir для серводрайверов. EXE симулирует завершение шагов Z/C виртуально.

```
  │
```

## 4. Единый протокол исполнения шага

Цикл одного шага гибки (EXE → ESP → PLC):

| № | Этап | Входные параметры | Условие начала | Условие завершения | Timeout | Код ошибки | Безопасная остановка |
|---|------|-------------------|----------------|-------------------|---------|------------|----------------------|
| 1 | **Feed Z** (Dry Run) | `L_feed` (мм) | `machine_state = ARMED` | EXE симулирует OK | 10 с | `ERR_Z_TIMEOUT` | SON=0, остановка |
| 2 | **Rotate C** (Dry Run) | `rotation` (°) | Feed завершён | EXE симулирует OK | 10 с | `ERR_C_TIMEOUT` | SON=0, остановка |
| 3 | **Bend** (PLC) | `angle` (°) | Rotate завершён; SON=1; нет аларм | PLC: угол достигнут (`INP`); Modbus reg=BEND_DONE | 30 с | `ERR_BEND_TIMEOUT` | PLC аварийная остановка |
| 4 | **Return bend** | — | Bend завершён | PLC: ролик вернулся в 0°; Modbus reg=READY | 15 с | `ERR_RETURN_TIMEOUT` | PLC стоп |
| 5 | **Clearance** | `ClampLength` (мм) | Return завершён | Dry Run пауза | 5 с | — | — |
| 6 | **Step complete** | — | Все этапы OK | EXE получает ACK; advance step_index | — | — | — |

## 5. State Machine шага (ESP8266)

```
IDLE
  │ (получена команда START_STEP от EXE)
  ▼
FEED_Z_DRY_RUN ──[timeout]──▶ ERROR
  │ (EXE симулирует OK)
  ▼
ROTATE_C_DRY_RUN ──[timeout]──▶ ERROR
  │ (EXE симулирует OK)
  ▼
WAIT_BEND_START
  │ (отправить PLC команду START_BEND)
  ▼
BEND_IN_PROGRESS ──[timeout / alarm]──▶ ERROR
  │ (Modbus PLC reg[BEND_STATUS] = DONE)
  ▼
WAIT_RETURN ──[timeout]──▶ ERROR
  │ (Modbus PLC reg[BEND_STATUS] = READY)
  ▼
CLEARANCE_DRY
  │
  ▼
STEP_ACK ──▶ отправить ACK в EXE ──▶ IDLE
```

При `PAUSE`: текущий этап заморожен, ответ `HOLD`.
При `STOP/E-STOP`: немедленно → `ERROR`, PLC → аварийная остановка.

## 6. Задачи по подсистемам

### 6.1 EXE — режим Machine Runtime

Добавить панель «Станок» (кнопки):

| Кнопка | Действие |
|--------|----------|
| `Connect` | HTTP GET к ESP8266, проверка связи, отображение статуса |
| `Home` | Команда домашней позиции на ESP/PLC |
| `Dry Run` | Запуск цикла с симуляцией всех осей (без реального движения) |
| `Start` | Запуск реального цикла (гибка через PLC, Z/C — Dry Run) |
| `Pause` | Остановка на текущем этапе шага, возобновление по Resume |
| `Stop` | Плановая остановка после текущего шага |
| `E-Stop` | Немедленная аварийная остановка (HTTP POST `/estop`) |
| `Resume` | Возобновление из HOLD |

**Поведение синхронизации:**
- Отправить шаг: `POST /step {step_index, feed, rotate, bend, uuid}`
- Ждать: `GET /status` → `machine_state = IDLE` + `ack_index == step_index`
- Продвинуть анимацию только после ACK
- При `alarm_code != 0` → пауза + отображение ошибки

**Лог событий:** таймштамп | событие | значение (сохранять в `logs/session_YYYYMMDD_HHMMSS.txt`)

### 6.2 ESP8266 — доработка прошивки

Файл: `ESP8266_Debug/ESP8266_Trubogib_Debug2/ESP8266_Trubogib_Debug2.ino`

**Расширить HTTP API:**

| Endpoint | Метод | Описание |
|----------|-------|----------|
| `/status` | GET | JSON: machine_state, step_index, ack_index, z_pos, c_pos, bend_angle, alarm_code |
| `/step` | POST | JSON: step_index, feed_mm, rotate_deg, bend_deg, uuid → ACK/NACK |
| `/pause` | POST | Перевести в HOLD |
| `/resume` | POST | Вернуть из HOLD |
| `/stop` | POST | Плановая остановка |
| `/estop` | POST | Аварийная остановка — немедленно |
| `/home` | POST | Команда Home |
| `/ping` | GET | Проверка связи |

**Расширить карту Modbus-регистров** (ESP как Master, PLC как Slave):

| Регистр | Запись/Чтение | Назначение |
|---------|---------------|------------|
| HR[0] | R | gCurrentAngle (текущий угол гибки, ×10) |
| HR[1] | R | gTargetAngle (целевой угол, ×10) |
| HR[2] | R | gState (статус ПЛК: 0=IDLE,1=BENDING,2=DONE,3=ERROR) |
| HR[3] | W | gCommand (0=STOP, 1=START_BEND, 2=RETURN) |
| HR[4] | W | gTargetAngleSet (установить целевой угол, ×10) |

**Требования к прошивке:**
- Неблокирующий main loop (без `delay()`)
- Watchdog связи с ПЛК: если нет ответа >3 с → перейти в ERROR
- Watchdog связи с EXE: если нет опроса >10 с → HOLD
- CRC-проверка Modbus-кадров; вести счётчик ошибок
- Безопасные статусы: при `alarm_code != 0` — запретить START_BEND
- Журнал ошибок: последние 20 записей в FIFO (endpoint `/log`)

### 6.3 PLC (CoDeSys) — доработка

Файл: `PLC110.30_m_v2/`

**State Machine:**

```
READY
  │ (HR[3] = START_BEND, HR[4] = targetAngle)
  ▼
BENDING ──[alarm / EMG]──▶ ERROR
  │ (angle >= targetAngle − 0.5°; INP подтверждение)
  ▼
BEND_DONE → HR[2] = 2
  │ (HR[3] = RETURN)
  ▼
RETURNING ──[timeout]──▶ ERROR
  │ (angle <= 0.5°)
  ▼
READY → HR[2] = 0
```

**Защиты:**
- Защита от двойного старта: если `BENDING` → игнорировать повторный START_BEND
- Защита очерёдности педалей в автоцикле
- При EMG: немедленно → ERROR, сбросить все выходы

### 6.4 Teensy — Roadmap (не реализовывать сейчас)

После поставки Teensy 4.1 переносится 1:1:
- HTTP API ESP8266 → HTTP API Teensy (те же endpoints)
- Modbus Slave (ID=1) → Modbus Slave (ID=1) (та же карта регистров, ПЛК = Master)
- Добавляется реальная генерация Step/Dir (RS-422, MAX490)
- Трапециевидный профиль скоростей (acc/dec)
- Процедура Home (поиск нулевого импульса энкодера)

## 10. Инструкция запуска комплекса (с нуля)

### Шаг 1 — Подготовка железа
1. Убедиться, что серво (обе оси) включены, CN2 подключён, JOG работает
2. Проверить питание CN1 (~25В, прямое подключение — согласно `02_Электрика_и_Схемы.md`)
3. Включить ПЛК, убедиться, что программа загружена и запущена

### Шаг 2 — Запуск Teensy 4.1 (заменяет ESP8266)
> **WiFi (ESP8266) больше не используется.** Использовался только для отладки по HTTP. Вся связь теперь через USB-Serial + Modbus RTU.

1. Подключить Teensy 4.1 по USB к ПК
2. Проверить: связь по Modbus RTU с ПЛК установлена (Serial1, 115200 8N1)
3. Проверить: EXE видит Teensy через USB-Serial

### Шаг 3 — Запуск EXE
1. Запустить `v2.1 (C++ OCCT)/build/Release/TubeBender.exe`
2. В панели «Станок» нажать `Connect`, указать IP `192.168.4.1`
3. Убедиться, что статус: `IDLE`, нет alarm_code

### Шаг 4 — Dry Run
1. Загрузить тестовую программу (Шаг 1: L=100, A=90°; Шаг 2: L=200, A=0°)
2. Нажать `Dry Run` — проверить синхронную анимацию без реального движения

### Шаг 5 — Реальный запуск
1. Убедиться, что ПЛК в состоянии `READY` (Modbus HR[2]=0)
2. Нажать `Start` в EXE
3. Наблюдать: анимация EXE синхронна с реальным движением оси гибки
4. Проверить лог на наличие ошибок

### Шаг 6 — Проверка аварийных сценариев
1. Нажать `E-Stop` во время цикла → система должна остановиться немедленно
2. Разорвать WiFi-соединение → watchdog ESP8266 должен перевести в HOLD
3. Сымитировать сработку концевика → ПЛК должен перейти в ERROR
